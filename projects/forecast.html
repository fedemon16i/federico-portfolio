DESIGN CONTEXT

Working inside Applaudo's design system

All four modules were built using Applaudo's existing component library. No new variants were introduced without engineering sign-off — keeping handoff friction low and implementation velocity high.

01

Existing components only

Used the component library as-is. Every design decision grounded in available patterns — no custom components without PM and engineering approval.

02

Design tokens applied consistently

Color, spacing, and typography tokens from the shared system applied throughout all modules — ensuring visual coherence across the platform.

03

State-driven patterns

Explicit states designed for every interaction — empty, loading, error, confirmation, and destructive action dialogs — following established system patterns.

04

Clean Figma handoff

Frames annotated with component references, spacing specs, and behavior notes — reducing back-and-forth with engineers during implementation.

05

Agile-paced iterations

Designs scoped to sprint capacity. Each module iterated across multiple cycles with direct feedback loops from PM, engineering, and stakeholder reviews.

MODULE 01

Pending Assignment Handling

Planners needed a clear way to manage assignments that weren't yet confirmed — without blocking the rest of the planning flow.

  • Differentiated confirmed and pending allocation states with clear visual hierarchy — planners always knew which assignments were locked vs. tentative.

  • Designed non-blocking flows — pending states did not freeze other planning actions, enabling parallel work across the workforce schedule.

  • Explicit confirmation step before converting a pending assignment to confirmed — preventing accidental state changes for operations teams managing real headcount.

  • All patterns scoped to Applaudo's existing design system — no new components introduced, keeping implementation overhead at zero.

MODULE 02

Timeframe Management

Planning teams needed to visualize allocation periods across overlapping project timelines — without losing track of who was available and when.

Timeframe Management — allocation period view

Clear period visualization

Start and end dates surfaced prominently for every allocation — planners can scan at a glance without drilling into individual records.

Overlapping assignment states

Designed explicit visual treatment for employees assigned to multiple projects simultaneously — making capacity conflicts immediately visible.

Sprint-aligned timeframes

Timeframe structure aligned with Agile sprint cycles — keeping allocation planning in sync with how engineering and PM teams already worked.

MODULE 03

Placeholder Management

Projects often need to reserve slots for roles not yet filled. Placeholders let planning continue without blocking downstream workflows on unfilled positions.

Placeholder Management — role reservation view

  • Reserve project slots by role, seniority, and technology stack — even before the actual employee is identified or assigned.

  • Placeholders participate in the same capacity and cost calculations as confirmed employees — giving planners an accurate picture even in an incomplete state.

  • Replace flow designed for minimal friction — converting a placeholder to a real employee assignment in a single action, carrying over all timeframe and project data.

MODULE 04

Employee Assignment Actions

Operations teams need to act on assignments constantly — editing dates, copying allocations, extending timeframes, and removing assignments. Every action carries risk; the design accounts for it.

  • Edit, copy, extend, and remove actions available on every assignment — surfaced contextually without cluttering the main planning view.

  • Destructive actions (remove, overwrite) gated behind explicit confirmation dialogs — preventing accidental data loss for teams managing live headcount.

  • Copy flow preserves all source assignment data (dates, hours, project) as editable defaults — reducing manual re-entry for recurring allocation patterns.

  • Validation logic inline — error states appear at field level, not after form submission, keeping planners oriented during complex multi-field edits.
