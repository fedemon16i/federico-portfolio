Problem Statement

Employees at Dollarama (Canada) and DollarCity (LATAM) stores were losing significant time on routine video-based transaction reviews. The existing SaaS platform wasn't delivering the efficiency stakeholders needed — creating operational drag and leaving fraud risks undetected.

A fraud monitoring and reporting feature had to be integrated while the platform was already in active development, with real users and real constraints. The challenge was not to redesign from scratch but to fit a high-complexity workflow into a system already in motion.

Primary Persona — The Telemarketer

A non-technical store employee responsible for reviewing video clips tied to POS transactions to flag potential fraud. Works across multiple tabs with no unified interface, often losing context between the video player and transaction list.

  • Goal: Complete daily video review quota without missing flagged transactions
  • Frustration: Has to open 3–4 browser tabs to cross-reference data
  • Need: Single-screen view combining POS search, video preview, and download state
  • Trust barrier: No feedback on whether a video is downloading or available without refreshing

Journey maps revealed where friction concentrated across the two main flows:

Flow A — POS → Video Review
Search POS terminal & date
Find transaction in results list
Open new tab for video player [friction]
Request video download
Refresh tab to check status [friction]
Play and review clip
Flow B — My Video Requests
Land on platform (no clear entry)
Navigate to request queue [buried]
Filter by status — no filter available [friction]
Open video — re-downloads if status unclear
Supervisor cannot see team queue
  • Users opened 3–4 browser tabs to cross-reference video and transaction data
  • No preview of video before committing to a download request
  • Scroll-heavy POS search forced users off-screen to reach video playback
  • No feedback state when videos were downloading or fully available
  • Filters were hidden behind secondary panels, causing missed refinements
  • Role-based views were not enforced — supervisors saw the same UI as telemarketers
  • "My Video Request" was the most-used entry point — yet it wasn't the default landing state

Based on research, we proposed three structural changes:

  • One Screen Rule — Consolidate POS search, video filter, and playback into a single scrollable panel. Eliminate all tab-switching.
  • Download State System — Introduce 4 explicit states (Not Downloaded → Downloading → Downloaded → Playing) with visual badges and toast notifications.
  • Role-Based Default Views — "My Video Requests" as default for telemarketers; global queue accessible by supervisors via role toggle.

Design Decisions

Three decisions that shaped the outcome

01
One Screen Rule

Kept POS search, video filter, and playback on one scrollable panel. Eliminated tab-switching that caused 60%+ of task friction in baseline testing. Users went from 3–4 open tabs to a single unified view without losing any functionality.

02
Download State System

Introduced 4 explicit video states — Not Downloaded → Downloading → Downloaded → Playing — with visual badges and a toast confirmation. Users could now trust the queue without polling, reducing duplicate requests to near zero in testing.

03
My Video Request as Default View

Surfaced the user's own request queue as the landing state of Video Investigation. Supervisors accessed a secondary global view via role-based toggle. This single decision accounted for the largest drop in time-to-first-action across all user types.
